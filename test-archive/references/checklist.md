# Checklist Reference

每轮测试开始前与结束前对照检查。

## 开始前（开工自检）

- [ ] 工作目录正确（`pwd` 与工程根目录一致）
- [ ] 已 `python -m test_archive list` 看过现有批次，确认是否需要新建
- [ ] 已确认本轮是"新建批次"还是"复用已有批次"（不清楚就问用户）
- [ ] 若用户要求部署：已问清服务器地址、账号、部署方式；**未臆造**
- [ ] 批次简述已规整：用下划线、不含空格/斜杠
- [ ] `wkq/NN_简述/{plan,data,report}/` 三个子目录都已创建
- [ ] 已渲染 `plan/test_plan.md`，并填充了用例编号、预期结果、环境说明
- [ ] **方案已发给用户确认**（除非用户明确"自动执行、无需确认"）

## 执行中

- [ ] 每条用例的过程产出都落到了 `data/`，文件名以 `caseNN_` 开头
- [ ] data/ 文件名都是小写字母/数字/下划线，无中文
- [ ] 同一条用例多次执行的产出用 `caseNN_runM_*` 区分
- [ ] 部署日志、环境信息等跨用例产出也有文件（`deploy.log`、`env.txt` 等）
- [ ] 失败用例已保留完整的请求/响应/堆栈，**不要只截短片段**

## 收尾（出报告前自检）

- [ ] 已渲染 `report/test_report.md`
- [ ] 执行总结的通过/失败/阻塞数量与下方明细表一致
- [ ] **每条用例的结论都引用了 `data/` 中的证据文件路径**
- [ ] 失败/阻塞用例有现象、可能原因、复现步骤
- [ ] 缺陷清单已编号（bug-01, bug-02…）
- [ ] 结论与建议已写明：是否可发布、是否需要回归
- [ ] 已向用户口头汇报整体结论与产出物路径
- [ ] （若多人共享服务器）已用 `backup-sync` 把 `wkq/` 推送到远程仓库

## 常见错误对照

| 错误现象 | 修正方法 |
|---|---|
| 报告里写了"详见日志"但没给路径 | 改为 `wkq/NN_简述/data/caseNN_output.log` |
| data/ 里有 `结果.json` 这种中文名 | 重命名为 `case01_result.json` |
| 方案和数据混放在 plan/ 下 | 把数据移到 data/，只留方案文档在 plan/ |
| 同一批次跑了三遍但都叫 `test_report.md` | 旧的另存为 `test_report_v1.md`、`test_report_v2.md`，最新版用 `test_report.md` |
| 编号跳到 100 后又新建 99 | 已存在 ≥ 100 时新建自动用三位，不要手改 |
| 部署细节写进 plan/ 的环境表 | 部署过程日志放 data/deploy.log，环境表只填结论性信息 |
