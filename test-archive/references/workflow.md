# Workflow Reference

完整工作流：从用户提出"测试一下这个工程"到归档出报告，分七步。

## 步骤 0 — 判定是否需要新建批次

读取用户意图：

| 用户说法 | 处理 |
|---|---|
| "测试一下这个工程" / "出个测试方案" / "部署后跑测试" | **新建**批次 |
| "继续跑之前的方案" / "补测上一轮" / "回归" | **复用**已有批次 |
| "整理一下测试报告" 但已有 data/ 与方案 | **复用**已有批次 |

判断不准时，向用户确认；**不要臆测**。

## 步骤 1 — 检查并创建批次

```bash
cd <工程根目录>           # 或测试工作目录
python -m test_archive list        # 先看现有批次，避免编号冲突
python -m test_archive new "<简述>"  # 例如: new "登录接口回归测试"
```

输出会给出 plan/ data/ report/ 三个绝对路径，后续落文件都用它们。

**编号规则**：从 1 开始，两位补零（01, 02…）；超过 99 时自动升三位。已存在三位编号时，新建的也用三位，避免 99 → 100 的位数跳变。

**简述命名**：中文短语或英文短语，词间用下划线。空格和斜杠会被拒绝（路径安全）。例：`登录接口回归测试`、`concurrent_pressure`。

## 步骤 2 — 部署源码（如用户要求）

> Skill 本身**不内置部署逻辑**——这是用户的工程/服务器，部署方式应由用户决定。

按以下顺序处理：

1. 若用户**未提供**服务器地址、账号、部署方式，**先问**：
   - 测试服务器地址、登录方式
   - 部署方式（git pull / rsync / docker / 已有脚本）
   - 是否需要清理旧部署
2. 用户给出后，按其方式部署；过程输出保存到 `data/deploy.log`。
3. 部署后做一次健康检查（端口/HTTP/CLI 自检），把结果追加到 `deploy.log`。

**禁止臆造服务器信息。** Skill 的职责是确保 wkq/ 结构，不是替用户决定部署架构。

## 步骤 3 — 写测试方案

```bash
python -m test_archive render <编号或简述> plan --subject "<测试主题>" --out test_plan.md
```

这会把 `templates/test_plan.md` 渲染到批次目录的 `plan/test_plan.md`，自动填充批次名、工作目录、生成时间。

随后**手动填充**：

- 测试目标与范围
- 用例列表（编号 case01, case02…，每条带预期结果）
- 测试环境说明（服务器、版本、依赖）
- 风险与回退

**写完一定要让用户确认**，除非用户明确表示"自动执行、无需确认"。

## 步骤 4 — 执行测试，落原始数据

按方案逐条执行。每条用例的过程产出保存到 `data/`，**文件名必须与用例编号对应**：

```
data/case01_output.log
data/case02_response.json
data/case02_screenshot.png
data/case03_stderr.log
data/deploy.log          # 若有部署步骤
```

文件名规范：小写字母 + 数字 + 下划线，可含 `.log` `.json` `.png` 等扩展。**不要用中文文件名**（目录名可中文，内部文件名英文/拼音，避免跨平台问题）。

## 步骤 5 — 生成测试报告

```bash
python -m test_archive render <编号或简述> report --subject "<测试主题>" --out test_report.md
```

随后**手动填充**：

- 执行总结：通过/失败/阻塞数量
- 用例结果明细表，**每条结论必须引用 `data/` 下的证据文件路径**
- 失败用例的现象、可能原因、复现步骤
- 缺陷清单
- 结论与建议（是否可发布、是否需回归）

## 步骤 6 — 收尾说明

向用户**简要**汇报：

- 整体结论（通过率、是否阻断）
- 关键缺陷（若有）
- 产出物路径，例如：
  ```
  方案: wkq/03_并发压测/plan/test_plan.md
  数据: wkq/03_并发压测/data/
  报告: wkq/03_并发压测/report/test_report.md
  ```

不要在对话里复述报告全文——用户能直接读文件。

## 步骤 7 — （可选）备份

如果工作目录跑在多人共享服务器上，可调用 [`backup-sync`](../backup-sync/SKILL.md) skill 把 `wkq/` 增量推送到 GitHub，防止他人误删。
